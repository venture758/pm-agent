## Context

当前代码虽然已经有数据库表承接大部分业务状态，但仍然保留了多条本地持久化路径：

- `pm_agent/database.py` 在未提供 `database_url` 时自动创建本地 SQLite：`sqlite:///.pm_agent_store/pm_agent.db`
- `pm_agent/storage.py` 通过 `LocalStateStore` 维护 agent 全局状态，并兼容迁移本地 `state.json`
- `pm_agent/workspace_store.py` 仍创建 `.pm_agent_store/workspaces/<id>/` 目录，兼容迁移本地 `workspace.json`
- 上传接口会把 Excel 文件落盘到 `uploads/` 目录，再把路径传给导入/同步逻辑

这意味着数据库不是唯一事实来源，且程序在数据库缺失时会悄悄退回到本地存储，与“所有持久化都来自数据库”的目标冲突。

## Goals / Non-Goals

**Goals:**
- 数据库成为系统唯一的持久化来源
- 应用在未提供数据库连接时明确失败，不再自动创建本地数据库
- 删除所有本地文件状态迁移和本地上传落盘逻辑
- 上传处理改为内存解析，最终只把业务数据和必要元数据写入数据库
- 更新测试和运行文档，明确数据库是必备前置条件

**Non-Goals:**
- 不重新设计现有数据库业务表结构
- 不改变已有业务 API 的核心语义
- 不在本次变更中引入对象存储、消息队列或新的外部基础设施
- 不保证兼容历史 `.pm_agent_store` 文件数据自动迁移到数据库

## Decisions

### 1. 启动策略：显式数据库配置是必选项

应用启动时必须传入 `database_url` 或提供等价环境变量；缺失时直接报错，而不是回退到本地 SQLite。

这样做的原因：
- 保证部署环境一致
- 避免“以为在用数据库，实际上数据写在本地文件”的隐性风险
- 符合用户“数据库是唯一存储”的要求

**Alternatives considered:**
- 保留本地 SQLite 作为开发环境 fallback：实现最省事，但与目标直接冲突
- 仅在生产环境强制数据库、开发环境保留 fallback：会继续保留双语义代码路径，后续问题仍然存在

### 2. 状态层：移除 `LocalStateStore` 的本地文件语义

`LocalStateStore` 的职责将被收敛为纯数据库状态访问层，或直接由 `DatabaseStore` 替代。无论保留类名还是重命名，其行为都不再包含：
- 创建本地根目录
- 读取/迁移 `state.json`
- 依赖本地路径作为状态来源

同样，`WorkspaceStore` 不再创建 `workspaces/<id>/` 目录，也不再扫描 `workspace.json` 进行兼容迁移。

### 3. 上传处理：不再落盘，改为内存解析

当前上传链路的核心问题不是“有没有数据库”，而是“原始 Excel 先写本地文件，再从路径读取”。本次改为：

- 上传接口直接接收 `bytes`
- Excel 导入函数支持 `bytes` / `BytesIO` 输入
- 同步/导入流程在内存中完成解析
- 最终只把标准化后的业务数据写入数据库
- 上传历史如需保留，仅保存元数据（文件名、时间、类型）到数据库中的工作区状态

**Alternatives considered:**
- 把原始 Excel 作为 BLOB 存入数据库：也满足“数据库唯一存储”，但会明显增加数据库体积和读写成本；当前需求并未要求保留原始文件重放能力
- 继续写入 `/tmp` 再解析：虽然不是业务持久化，但仍保留文件路径依赖，违背“移除本地存储方式”的方向

### 4. 兼容策略：不再自动迁移本地文件数据

本次变更明确去掉：
- `state.json` 自动迁移
- `workspace.json` 自动迁移

理由：
- 用户已经明确数据全部来自数据库
- 自动迁移逻辑会保留本地扫描和路径依赖
- 一旦保留兼容迁移，代码中仍然需要维护本地文件语义

这意味着历史本地目录不再被系统读取；若确有一次性导库需求，应通过独立脚本处理，而不是保留在运行时主链路中。

## Risks / Trade-offs

- **[风险]** 未配置数据库时应用将直接启动失败 → **缓解**：在配置和 README 中明确数据库为必需项，并在启动时报出可操作错误信息
- **[风险]** Excel 导入函数从路径改为内存流后，调用面较多 → **缓解**：统一把导入函数签名升级为同时接受 `bytes` / 文件对象，逐条回归测试上传链路
- **[风险]** 移除本地迁移后，旧 `.pm_agent_store` 数据不再自动可见 → **缓解**：提供独立迁移说明，明确该数据不再属于运行时兼容范围
- **[权衡]** 不持久化原始上传文件会失去“原文件重放”能力 → **有意为之**：当前目标是消除本地持久化与路径依赖，而不是扩张数据库二进制归档范围

## Migration Plan

1. 删除默认本地数据库配置与本地目录常量的运行时依赖
2. 将 agent/workspace 状态访问层改为纯数据库模式
3. 重构上传解析入口，移除本地 `save_upload(...stored_path...)` 链路
4. 清理 `state.json` / `workspace.json` 自动迁移逻辑
5. 更新测试、README 和启动说明

回滚策略：
- 若变更需要回滚，只能恢复当前本地 fallback 代码路径
- 由于本次目标是“删除本地路径”，回滚应以代码版本回退实现，而不是运行时开关切换

## Open Questions

- 上传历史是否需要在数据库中保留原始文件二进制，还是仅保留元数据即可
- 是否需要提供一次性的离线迁移脚本，把现有 `.pm_agent_store` 中的本地数据导入用户数据库
