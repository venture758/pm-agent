## Context

当前 Java 后端的需求解析链路已具备 LLM 调用与结构化返回能力，但输出重点仍在需求标题与基础字段，缺少稳定的模块归属结果。现有系统同时持有两类可用于语义定位的上下文资产：
- 业务模块知识（`big_module`、`function_module`、负责人等）
- 历史任务数据（任务名称、关联用户故事等文本字段）

如果不在解析阶段统一产出模块归属与抽象语义，后续推荐与人工确认会持续依赖手工补录，导致效率和一致性下降。

## Goals / Non-Goals

**Goals:**
- 在需求解析结果中稳定输出 `big_module`、`function_module`。
- 在 LLM 提示词中融合业务模块与历史任务名称上下文，提高模块匹配质量。
- 禁止把历史 `module_path` 作为模块归属的核心信号，避免引入脏数据误导。
- 输出“抽象提炼”字段与匹配依据，支持人工审查和可追溯性。
- 为低置信或未命中场景提供显式降级结果（候选模块/待确认状态）。

**Non-Goals:**
- 不在本次变更中改造推荐打分算法本身。
- 不引入新的外部向量检索服务或全文检索中间件。
- 不重构历史任务存储模型，仅复用现有可查询字段。

## Decisions

### 1) 解析输出结构扩展为“归属 + 抽象 + 证据”
- 决策：在每条 requirement 结果中新增模块与语义字段：`big_module`、`function_module`、`abstract_summary`、`match_evidence`、`match_status`。
- 原因：后续流程需要直接消费模块归属，且必须可解释、可人工复核。
- 备选：仅输出 `big_module/function_module`，不返回依据。  
  放弃原因：难以判断误匹配来源，影响确认效率。

### 2) 上下文构建采用“模块候选摘要 + 历史任务样本”
- 决策：在调用 LLM 前构建受控上下文：
  - 模块侧：按 `big_module/function_module` 输出去重后的模块清单与关键词摘要
  - 任务侧：按最近窗口提取历史任务名称/故事名称样本，并做去噪与长度限制
  - 排除侧：历史 `module_path` 字段不进入匹配上下文，也不参与规则打分
- 原因：控制 token 成本并减少噪声，提高模型选择稳定性。
- 备选：直接拼接全量模块与全量任务。  
  放弃原因：上下文过长、噪声高，容易降低输出一致性。

### 3) 明确低置信降级协议
- 决策：当模型无法可靠匹配时，`match_status` 返回 `needs_confirmation`，并给出最多 N 个候选模块与理由；禁止伪造确定匹配。
- 原因：避免“看似正确”的错误模块进入后续流程。
- 备选：始终返回一个最佳猜测模块。  
  放弃原因：误匹配不可见，后续修正成本更高。

### 4) 兼容现有 API 契约演进
- 决策：新增字段按向后兼容方式扩展到 `/api/v2/workspaces/{workspaceId}/chat` 等解析响应；旧字段保持不变。
- 原因：减少前端联调冲击，支持灰度展示新字段。
- 备选：替换原有 requirement 结构。  
  放弃原因：改动面过大，风险高。

## Risks / Trade-offs

- [风险] 历史任务名称质量参差导致误导匹配  
  → Mitigation: 任务样本去噪（空值、模板词、重复词），并对 evidence 记录来源。
- [风险] 实现中误用 `module_path` 导致匹配偏移  
  → Mitigation: 在上下文构建层硬编码排除 `module_path`，并添加单测断言其不被注入。
- [风险] 新字段增加前端适配成本  
  → Mitigation: 保持兼容响应，前端按可选字段逐步启用展示。
- [风险] LLM 输出字段不完整或格式漂移  
  → Mitigation: 增强解析校验与默认值回填，未命中时进入 `needs_confirmation` 降级。
- [风险] 上下文增大导致延迟上升  
  → Mitigation: 限制模块与任务样本条数，按优先级截断上下文。

## Migration Plan

1. 后端先发布：新增字段解析、上下文构建、降级协议与响应扩展（保持向后兼容）。
2. 前端灰度展示：先在需求详情/确认界面显示模块归属与抽象提炼，不阻断既有流程。
3. 联调验证：覆盖高命中、低置信、空历史任务三类场景，以及“存在错误 `module_path` 但不影响结果”场景。
4. 回滚策略：出现严重误匹配时，可通过配置关闭“历史任务上下文注入”，保留原解析流程。

## Open Questions

- 历史任务样本窗口应取“最近 N 天”还是“最近 N 条”更稳定？
- `match_evidence` 是否需要标准化结构（关键词命中、任务命中、模型理由）以便后续统计？
- `abstract_summary` 的长度限制与展示位置是否由前端统一裁剪？
