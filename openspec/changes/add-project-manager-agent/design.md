## 背景

当前仓库仅初始化了 OpenSpec 配置，尚未实现任何 Agent 能力，因此本次变更既定义产品能力，也需要建立最小可扩展的项目经理 Agent 架构。该 Agent 面向软件研发团队，输入核心包括产品经理在群里发送的需求列表、团队成员情况、项目管理平台每天人工导出的故事和任务 Excel 数据，以及业务模块划分与成员熟练度 Excel 数据，输出必须是结构化、可执行、可复核的需求分配结果，而不是泛化建议。

项目经理 Agent 的核心价值不在于替代所有人类决策，而在于稳定完成五类高频动作：群聊需求整理与分析、分配推荐与平台落地、执行监控、团队负载与能力管理、模块知识库维护。设计上需要优先保证分配依据透明、规则稳定、异常显式暴露，同时保留后续接入外部系统的扩展点。

## 目标 / 非目标

**目标：**

- 提供统一的项目经理 Agent 工作流，覆盖群聊需求接收、成员建模、分配推荐、人工确认、故事/任务落地、执行监控、资源洞察和模块知识库维护。
- 采用结构化领域模型表示需求特征、成员能力、模块归属、熟悉度、可用容量、分配结果、故事/任务映射、执行异常和能力风险，避免纯自然语言输出难以复用。
- 将消息解析、输入建模、模块知识库导入、匹配计算、确认操作、Excel 导入同步、监控分析和管理输出拆成清晰节点，便于测试、迭代和后续接入外部工具。
- 为推荐结果、确认结果、故事/任务结果、预警结果、知识库更新结果和负载洞察定义校验规则，确保核心字段完整可追踪。

**非目标：**

- 本次不直接实现与 Jira、飞书、GitHub、日历或 IM 等外部系统的深度双向集成。
- 本次不尝试做复杂的长期资源排期优化或跨项目全局调度。
- 本次不替代团队负责人最终确认，只输出建议和可执行草案。
- 本次不依赖项目管理平台的在线查询接口，平台数据以人工导出的 Excel 为准。

## 关键设计决策

### 1. 按现有团队流程和知识维护流程拆分工作流

决策：将项目经理 Agent 设计为由七个显式阶段组成的单 Agent 系统：`group requirement intake`、`module knowledge sync`、`demand analysis`、`assignment recommendation`、`assignment confirmation`、`excel-based story/task sync`、`execution monitoring and team insight`。

原因：

- 用户给出的真实流程覆盖群聊评审、组长回复、平台录入、每日 Excel 导出和执行跟踪，此外还存在模块负责人和熟练度维护需求，按流程阶段拆分比单一“大提示词”更容易控制结果质量。
- 每个阶段可绑定独立输入模式、输出 schema 和校验逻辑，降低提示漂移风险。
- 后续若需要拆分为多个协同 agent，也可以沿相同边界演进。

备选方案：

- 单提示词统一处理所有场景：实现快，但难以约束输出质量，测试粒度也过粗。
- 多 agent 自主协作：扩展性强，但当前仓库尚无基础设施，复杂度过高。

### 2. 建立统一的评审到交付状态模型

决策：以统一的状态对象作为所有阶段的输入输出中心，至少包含需求项、群消息原文、优先级、类型、复杂度、风险、技能标签、截止信息、成员角色、技能、当前负载、可用容量、模块知识库条目、分配结果、确认操作、故事记录、任务记录、导入批次、编码映射、执行信号、预警结果和能力风险说明。

原因：

- 需求分析、模块匹配、分配推荐、人工确认、平台落地和执行监控依赖同一事实基础，统一状态模型可以避免每个阶段重复解析上下文。
- 有利于后续接入持久化层、工具调用层和测试夹具。
- 可以在 Agent 输出前做 schema 校验，确保字段完整。

备选方案：

- 每次基于原始文本重新推理：实现简单，但难以保持状态连续性。
- 为每个阶段维护独立数据结构：局部灵活，但会增加映射和一致性成本。

### 3. 将推荐分配建立在模块知识库之上

决策：将“模块知识库”作为分配推荐的主要依据之一。Agent 先结合模块负责人、B角、成员熟悉度和当前负载生成负责人、备选人、测试人员和拆分建议，再记录项目经理对采纳、改派、拆分和协作人的确认结果，并在导入 Excel 时按 `用户故事编码` 和 `任务编号` 执行故事和任务的新增或更新。

原因：

- 实际流程中开发组长和测试组长会显式参考模块负责人、B角和熟悉度来做判断，平台数据也只能通过人工导出回流，因此需要把知识库、推荐、确认和系统落地清晰分层。
- 规则校验可检查成员过载、技能缺口、重复分配、高优需求无人承接、测试遗漏、拆分后的责任遗漏，以及编码冲突导致的重复数据问题。
- 验证层便于通过单元测试覆盖，不必依赖模型稳定性解决所有问题。

备选方案：

- 完全依赖提示词约束：成本低，但对复杂输入的稳健性不足。
- 人工后处理：质量可控，但失去自动化价值。

### 4. 统一知识库、平台落地与监控输出结构

决策：为分配结果、模块知识库同步结果、故事/任务落地结果、Excel 导入同步结果、执行预警和负载洞察分别定义统一输出 schema，至少覆盖需求标识、命中模块、推荐负责人、备选负责人、测试负责人、分配理由、确认动作、故事字段映射、任务字段映射、同步动作类型、编码键、风险级别、负载热度、单点依赖和建议动作。

原因：

- 结构化输出是接入群回复草稿、模块知识库维护、故事/任务 Excel 导入、预警和管理报表的前提。
- 统一 schema 便于测试直接校验关键字段，而不是解析自由文本。

备选方案：

- 仅输出自然语言建议：实现最简单，但难以复核和程序化消费。

## 风险与权衡

- [输入上下文不完整导致分配失真] → 在 intake 流程中显式输出缺失字段和默认假设，禁止在技能或容量未知时伪装精确分配。
- [群消息格式不稳定导致解析偏差] → 需求列表和分配回复都保留原文，同时将解析结果标记置信度和待人工确认字段。
- [模块知识库表结构复杂且年度版本并存] → 默认优先使用最新年度 sheet，并将 sheet 名称、版本和导入时间纳入知识库元数据。
- [成员负载数据不准导致分配偏差] → 将当前负载和可用容量作为显式输入字段，没有数据时标记低置信度。
- [技能匹配判断依赖主观推理] → 采用明确的匹配维度，如角色、技术栈、领域经验和可用容量，并要求输出分配理由。
- [执行监控信号来源不稳定] → 将延期、阻塞和质量异常建模为显式输入事件，缺失时只输出有限预警。
- [平台导出字段与群聊字段不一致] → 在故事/任务映射层建立标准字段模型，并允许原始字段透传用于追溯。
- [每日导入时出现重复或漏同步] → 以 `用户故事编码` 和 `任务编号` 作为幂等键，记录导入批次和同步动作，确保同一编码只新增一次，后续走更新。
- [知识库更新可能覆盖人工判断] → 自动更新只调整分配历史、最近命中模块和熟悉度建议值，对主负责人和 B 角采用“建议变更待确认”模式。
- [能力管理结论过度推断] → 单点依赖和培养建议必须基于历史分配、技能覆盖和负载事实，不输出无依据判断。
- [结构化输出增加实现复杂度] → 先定义最小可用 schema，只覆盖核心字段，后续按集成需要扩展。
- [后续工具集成边界不清] → 在工作流层预留 tool adapter 接口，但本次不实现真实适配器。
