## Why

当前知识更新 Agent 虽然已经有运行状态和摘要，但业务模块在自动更新前后的具体变化没有被单独记录，导致无法核对 Agent 实际改了什么、是否改对、以及一次更新影响了哪些模块。现在需要把更新前快照、更新后结果和差异记录独立出来，形成可观察、可追踪的审计链路。

## What Changes

- 为知识更新 Agent 增加“业务模块更新前记录”和“业务模块更新后记录”两类独立留痕。
- 在一次知识更新执行内，按 `workspace_id + session_id + module_key` 保存模块更新前快照、更新后快照和字段级差异摘要。
- 扩展工作区/确认历史相关查询接口，返回最近一次知识更新涉及的模块变更明细与前后状态摘要。
- 在确认中心或确认历史视图中展示知识更新前后记录，方便人工观察和核对 Agent 自动更新结果。
- 明确知识更新失败、跳过、无字段变化时的记录语义，避免“有运行状态但看不到变更内容”的灰区。

## Capabilities

### New Capabilities
- `knowledge-update-module-diff-records`: 记录并查询知识更新 Agent 对业务模块自动更新前后的独立快照与差异明细。

### Modified Capabilities
- None.

## Impact

- 后端模型与存储：知识更新记录模型、业务模块相关持久化表、工作区/确认历史查询聚合。
- API：确认提交后的返回 payload、确认历史接口、最近一次知识更新摘要接口。
- 前端：确认中心最近知识更新卡片、确认历史详情展示。
- 测试：知识更新成功/跳过/失败/无变更场景下的记录与展示回归。
