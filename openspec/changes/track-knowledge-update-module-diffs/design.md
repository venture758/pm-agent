## Context

当前知识更新 Agent 已经会在确认提交后运行，并把 `success / skipped / failed` 状态记录到知识更新记录中，但记录粒度停留在“本次运行摘要”。业务模块是否被自动改写、改写前是什么状态、改写后变成什么状态、具体变更了哪些字段，当前没有单独存储和展示。

这个缺口直接影响两类使用者：
- 产品/项目经理无法核对 Agent 自动更新是否符合预期。
- 开发/运维无法定位一次知识更新到底影响了哪些模块和字段。

约束如下：
- 现有知识更新能力已经基于 `workspace_id + session_id` 形成运行记录，新增能力应复用这个上下文，不引入新的触发入口。
- 自动更新对象是业务模块条目，因此记录粒度必须落到 `module_key`，而不是只记录会话级摘要。
- 需要兼容 `success / skipped / failed` 语义，避免失败或跳过时前端误以为“没有数据就是没有运行”。

## Goals / Non-Goals

**Goals:**
- 为每次知识更新运行生成业务模块级别的更新前快照、更新后快照和字段差异摘要。
- 支持按 `workspace_id + session_id` 查询本次知识更新涉及的全部模块变更记录。
- 在最近一次知识更新摘要与确认历史详情中展示模块变更结果，便于人工观察。
- 明确无字段变化、跳过、失败三种情况下的记录策略，使查询结果可解释。

**Non-Goals:**
- 不改变知识更新 Agent 的触发时机和核心 LLM 提示词。
- 不在本次变更中引入“人工审核后再落库”的审批流程。
- 不扩展到人员画像、任务、故事等非业务模块对象的前后差异记录。
- 不做复杂 diff 可视化编辑器，只提供结构化快照和差异摘要展示。

## Decisions

### 1. 增加独立的模块变更记录表，而不是把前后快照塞回 `KnowledgeUpdateRecord`
- 决策：新增独立持久化模型，用 `workspace_id + session_id + module_key + stage` 或等价结构保存模块更新前后记录，并保留字段级差异摘要。
- 原因：知识更新会话级记录和模块级变更明细是两种粒度，混在同一 JSON 中会导致历史查询、分页、前端渲染和后续排障都变重。
- 备选方案：把全部前后快照直接嵌入现有 `workspace_knowledge_update_records.payload_json`。
- 不采用原因：单条 payload 会迅速膨胀，不利于按模块查询，也不利于未来单独展示或导出。

### 2. 记录“before / after / diff_summary”三类信息，`after` 以后置持久化结果为准
- 决策：
  - `before_snapshot` 取自动更新执行前、落库前的模块原始状态。
  - `after_snapshot` 取自动更新应用完成后的最终状态。
  - `diff_summary` 以字段维度列出新增、删除、修改项。
- 原因：只有把前后快照和差异摘要一起保留，前端才能既看结果，也能快速知道改动点。
- 备选方案：只记录 diff，不保存完整快照。
- 不采用原因：当 diff 逻辑出错或展示需求变化时，缺少完整快照会失去核对依据。

### 3. 记录粒度按“受影响模块”生成，而不是全量模块快照
- 决策：仅为本次知识更新实际命中的 `module_key` 生成前后记录。
- 原因：全量模块快照成本高、噪音大，而且不符合“观察本次自动更新影响范围”的目标。
- 备选方案：每次知识更新都对整个工作区模块知识库做全量前后快照。
- 不采用原因：会显著放大存储和查询成本，也降低人工查看效率。

### 4. 无字段变化也保留记录，但显式标记 `changed=false`
- 决策：当知识更新 Agent 命中了某个模块，但应用更新后字段值没有变化时，仍然生成模块级记录，并在差异摘要中标记无变化。
- 原因：这类场景对用户同样重要，能区分“Agent 没处理这个模块”和“Agent 处理了，但没有实际改动”。
- 备选方案：无变化时不写任何模块记录。
- 不采用原因：会让观察者误判为链路未执行或命中范围不明。

### 5. `skipped / failed` 不生成伪造的 after 快照，但会保留会话级可解释状态
- 决策：
  - `skipped` 和 `failed` 仍然保留会话级知识更新记录。
  - 只有当系统实际进入模块更新计算阶段时才生成模块级 before/after 记录。
  - 如果在生成 diff 前失败，可记录错误摘要但不伪造 after 数据。
- 原因：需要保证审计数据真实，不把“未更新”包装成“更新为空”。
- 备选方案：在失败时也写空的 before/after 记录占位。
- 不采用原因：会污染前端展示语义，降低排障价值。

### 6. 查询出口沿用现有确认中心链路，前端不新增一级入口
- 决策：在现有“确认中心”的最近一次知识更新卡片和确认历史详情中增加模块变更明细入口或展开区，不新增独立菜单。
- 原因：用户观察这个信息的上下文就是“某次确认触发的知识更新”。
- 备选方案：新增独立“知识更新审计”页面。
- 不采用原因：信息架构会分散，当前需求也不要求独立运营页面。

## Risks / Trade-offs

- [模块快照 payload 变大] → 只记录受影响模块，避免工作区全量快照；前端默认展示摘要，详情按需展开。
- [diff 计算规则不稳定] → 先限定为结构化字段级 diff，按列表/标量/对象三类规则归一化输出。
- [失败时记录不完整引发误解] → 会话级状态必须带错误原因；模块级记录仅在真正进入更新阶段后生成。
- [前端信息过载] → 最近一次知识更新卡片只显示摘要和受影响模块数，详细前后记录放到历史详情或折叠面板。

## Migration Plan

1. 新增模块级知识更新前后记录模型与存储表。
2. 在知识更新执行链路中，先采集受影响模块 `before_snapshot`，应用更新后生成 `after_snapshot` 和 `diff_summary`，再统一落库。
3. 扩展确认提交返回 payload 与确认历史查询结果，附带最近一次或对应 session 的模块变更摘要。
4. 前端确认中心读取新增字段，展示最近一次更新的模块变更概览，并在历史详情中展示前后记录。
5. 增补成功、无变化、跳过、失败场景测试。

回滚策略：
- 回滚应用版本后可以保留新增表和历史记录，不影响旧逻辑读取。
- 如需临时关闭展示，可前端隐藏模块差异面板，后端继续保留数据写入。

## Open Questions

- 模块级明细在确认历史中默认展开还是按需展开，可在实现时根据现有页面密度决定；本次先要求“可查看”，不强制默认展开。
