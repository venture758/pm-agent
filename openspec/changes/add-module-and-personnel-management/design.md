## Context

当前项目经理工作台已经具备需求录入、模块知识库 Excel 导入、推荐确认、同步监控和团队洞察能力，但“业务模块”和“人员”两类基础数据仍缺少稳定的维护入口。模块数据主要依赖 Excel 导入写入 `AgentState.module_entries`，人员数据在现状下仍会混入需求处理阶段的 `team_rows` 临时表格；这导致基础数据和单次需求处理过程耦合过深，用户无法把模块和人员信息当作长期资产持续维护。

这次变更要补齐的是维护能力，而不是重写推荐算法。系统需要提供明确的页面入口、API 边界和持久化方式，让项目经理可以单独维护业务模块与人员信息，并让这些维护结果直接成为后续推荐、监控和洞察的事实来源。

## Goals / Non-Goals

**Goals:**

- 提供独立的业务模块维护能力，支持模块条目查询、新增、编辑、删除和基础校验。
- 提供独立的人员管理维护能力，支持成员画像查询、新增、编辑、删除和基础校验。
- 让维护后的模块和人员数据被推荐、监控和团队洞察流程直接复用，并去掉 intake 临时成员来源。
- 保留现有 Excel 导入能力，使其可以作为批量导入入口与页面维护入口共存。

**Non-Goals:**

- 本次不引入组织级权限、审批流或多人并发冲突解决机制。
- 本次不重做现有推荐、确认、同步、监控和洞察算法。
- 本次不实现复杂批量编辑器、导入模板设计器或审计报表。
- 本次不要求跨工作区共享统一主数据，默认仍以工作区维度维护。

## Decisions

### 1. 为模块和人员维护提供独立页面入口，而不是继续塞进需求输入页

决策：在现有工作台中新增独立的“业务模块维护”和“人员管理”页面或等价导航入口，而不是继续扩展 `IntakeView` 的现有表单。

原因：

- 当前需求输入页已经承载群消息录入、结构化需求、团队临时编辑和知识库导入，再继续叠加维护功能会让信息架构失控。
- 维护功能是长期主数据管理，不应与单次需求 intake 混在同一块交互里。
- 独立页面更便于后续增加筛选、搜索、校验提示和批量操作。

备选方案：

- 把维护功能继续做成 intake 页面中的一个区块：实现初期更快，但会让“处理需求”和“维护主数据”两个目标混杂。

### 2. 模块维护直接以 `AgentState.module_entries` 为后端事实来源

决策：业务模块维护 API 直接围绕 `AgentState.module_entries` 读写，页面展示与保存都操作同一套模块知识条目结构。

原因：

- 当前推荐、洞察和知识库摘要已经依赖 `module_entries`，直接复用可以避免复制一份平行状态。
- Excel 导入本来就写入这套结构，手工维护与导入维护共享同一事实来源更自然。
- 这样可以把“导入”和“维护”统一成同一知识库的两种更新方式。

备选方案：

- 单独维护一份前端专用模块表，再同步到 `module_entries`：会增加状态分叉和同步风险。

### 3. 人员管理引入独立的工作区持久化成员集合，并作为唯一人员来源

决策：为人员管理引入工作区级的持久化成员集合，推荐、监控和洞察统一读取这份维护后的成员数据，不再接受 intake 阶段的临时成员覆盖。

原因：

- 当前 `draft.team_rows` 更像单次需求处理的编辑态，不适合作为长期人员主数据。
- 人员管理需要支持长期维护角色、技能、容量和约束，并在页面刷新后继续可用。
- 把维护成员设为唯一来源后，可以避免系统同时维护两套人员入口和取数优先级。

备选方案：

- 保留 `draft.team_rows` 作为临时覆盖来源：短期更灵活，但会让人员主数据再次分叉，并增加推荐结果不可追溯的风险。

### 4. API 采用资源化读写接口，页面保存以“整体列表 + 单项编辑”模式为主

决策：后端为业务模块和人员管理提供清晰的资源化接口，支持列表读取、创建、更新和删除；前端初版以表格/卡片列表加单项编辑为主，而不是一次性提交复杂嵌套表单。

原因：

- 维护场景天然适合资源化边界，便于前后端测试和后续扩展。
- 单项编辑和列表展示更易插入字段校验、重复检查和删除确认。
- 这类边界也更适合后续引入搜索、筛选和批量导入。

备选方案：

- 只提供一个大表单批量保存所有成员/模块：实现简单，但错误定位和冲突提示会更差。

## Risks / Trade-offs

- [移除 intake 临时成员后，老使用方式需要调整] → 在页面和文档中明确人员维护页是唯一入口，并在推荐前对空人员库给出清晰提示。
- [模块手工维护与 Excel 导入同时存在可能造成覆盖] → 统一写入同一知识库结构，并在界面展示来源、更新时间或覆盖结果。
- [维护入口增加页面和 API 复杂度] → 采用资源化接口和独立页面，先覆盖核心 CRUD 与校验，避免首版过度设计。
- [字段校验不充分会破坏推荐质量] → 对空名称、重复键、负容量、非法角色和无效熟悉度做显式校验并阻止保存。

## Migration Plan

1. 扩展工作区状态与后端服务，明确模块条目和维护成员的持久化读写方式。
2. 新增业务模块和人员管理相关 API，并把推荐/洞察流程接入新的唯一成员来源。
3. 在前端增加对应导航与维护页面，完成列表展示、编辑、删除和校验反馈。
4. 移除 `team_rows` 等 intake 临时成员取数路径，并补充测试覆盖新的唯一来源规则。
5. 更新文档，说明页面维护与 Excel 导入的适用场景，以及人员维护页是唯一成员入口。

回滚策略：

- 若维护入口实现不稳定，可下线新页面和对应 API，现有 Excel 导入和需求处理流程仍可继续使用。
- 持久化结构应保持向后兼容，避免影响已有工作区数据读取。

## Open Questions

- 首版人员管理是否需要支持“在职/离岗”状态，还是先只做删除与编辑。
- 模块维护中成员熟悉度是自由文本、枚举值还是分级数值，是否需要与现有导入格式完全对齐。
- 是否需要首版就支持从模块维护页直接引用已维护成员，还是先允许手工输入成员姓名。
