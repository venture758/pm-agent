## Context

当前交付页仅支持“故事+任务”双文件同步，故事管理页没有独立上传入口。后端也仅将故事导入到内存/状态快照结构，缺少面向故事 Excel 原始字段的结构化持久化表。

本次变更需要在已有架构（`frontend` + `pm_agent/api.py` + `pm_agent/api_service.py` + `DatabaseStore`）上增加一条独立的故事导入链路，并确保数据库模型覆盖 Excel 模板字段。样例文件来自 `docs/导出列表_用户故事列表_0410191819.xlsx`（表头 45 列）。

相关角色：项目经理（上传与查看导入结果）、开发（维护字段映射与模型）、测试（验证字段映射、幂等、异常处理）。

## Goals / Non-Goals

**Goals:**
- 在故事管理页提供单独上传用户故事 Excel 的能力，不依赖任务文件。
- 新增数据库故事表，字段覆盖模板 Excel 表头（45 列），支持按用户故事编码幂等 upsert。
- 返回导入统计与错误明细，便于用户快速定位格式问题。
- 保持现有“故事+任务同步”能力可用，不引入行为回归。

**Non-Goals:**
- 不在本次实现故事字段的在线编辑回写数据库。
- 不引入“任意 Excel 动态建表”能力；仅支持约定模板字段。
- 不改造任务导入模型与任务管理流程。

## Decisions

### 1) 新增独立故事上传接口，而不是复用双文件接口
- 决策：新增 `POST /api/workspaces/{workspace_id}/uploads/story-only`（命名可在实现时微调），仅接收 `story_file`。
- 理由：当前双文件接口有强耦合校验（必须同时存在 story_file/task_file），不适合单独故事上传。
- 备选方案：扩展原接口支持可选 task_file。
  - 未选原因：会让原语义变复杂，兼容分支多，影响已有调用和测试。

### 2) 使用“固定字段映射 + 显式列模型”，不做动态列创建
- 决策：根据模板表头建立固定数据库模型（45 列 + 元信息列），并在代码中维护 `EXCEL_HEADER -> db_column` 映射。
- 理由：SQL DDL、查询和索引都要求稳定列定义；动态建表/动态加列会显著提高迁移和运维复杂度。
- 备选方案：单 JSON 列存整行。
  - 未选原因：难以做字段级查询、筛选和约束，不满足“按照字段建数据库模型”的目标。

### 3) 以“用户故事编码”作为业务主键做 upsert
- 决策：`workspace_id + user_story_code` 唯一约束，导入时执行“存在更新，不存在新增”。
- 理由：用户故事编码是业务侧稳定键，适合作为幂等导入主键。
- 备选方案：按“用户故事名称 + 创建时间”去重。
  - 未选原因：标题可改且不唯一，去重不稳定。

### 4) 表头严格校验，允许受控别名
- 决策：默认要求模板列完整匹配；对已知小差异（例如空格、全角/半角）做规范化匹配；缺失关键列直接失败。
- 理由：减少误导入风险，避免列错位导致数据污染。
- 备选方案：只按列序解析不校验列名。
  - 未选原因：平台导出版本变化时风险高。

### 5) 迁移策略采用“先建表后接入”，保留原状态快照
- 决策：新增 `workspace_story_records`（MySQL + SQLite），原 `workspace.handoff_stories` 继续保留用于兼容展示；新链路优先从数据库读。
- 理由：避免一次性重构全部读取路径，降低回归风险。
- 备选方案：直接替换所有 story 数据来源为新表。
  - 未选原因：改动面大，难以在单次迭代稳定上线。

## Risks / Trade-offs

- [风险] 模板列名变更导致导入失败率上升
  → Mitigation: 维护列名别名表，导入响应给出“缺失列/未知列”详情。
- [风险] 45 列映射实现中出现字段错配
  → Mitigation: 增加字段映射单测（头部->列名、类型转换、空值处理）和端到端导入测试。
- [风险] 新老数据源并存造成展示不一致
  → Mitigation: 明确读取优先级（新表优先），并在导入成功后同步刷新故事管理页数据源。
- [权衡] 固定模型提升可维护性，但对模板变更的弹性降低
  → Mitigation: 通过迁移脚本与版本化映射逐步演进字段。

## Migration Plan

1. 新增数据库表 `workspace_story_records` 的 DDL（MySQL）和 SQLite schema 创建语句。
2. 增加后端解析器：读取故事 Excel，执行表头校验、类型转换与 upsert。
3. 增加独立上传 API 与 service 方法，返回导入摘要（总数/新增/更新/失败）。
4. 前端故事管理页新增“单独上传故事 Excel”交互与结果提示。
5. 补充测试：API 测试、导入解析测试、字段映射测试、前端 store/API 集成测试。
6. 灰度发布后观察导入失败原因分布，再补齐必要别名与容错。

回滚策略：
- 接口层回滚到旧版本，不再暴露 story-only 上传入口。
- 保留新增表不删除（避免数据丢失）；读取路径回退到旧快照结构。
