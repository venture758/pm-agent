## Context

当前前端工作台由一个全局壳层页面和多个业务视图组成。首页壳层在 `App.vue` 中渲染了 `metric-grid` 汇总卡片，各业务页又在首屏面板中分别渲染 `mini-stat-grid` 数字摘要；除此之外，壳层还会在正文前统一插入一个 `MessagePanel`“工作区消息”模块。它们叠加后，使用户进入页面时首先看到的是摘要与系统提示，而不是当前页的主操作或主要内容。

这次变更只涉及前端展示层，不涉及后端 API、工作区状态持久化或业务计算逻辑。实现重点在于两件事：一是统一识别并移除各页面首屏统计卡片；二是在移除后收口相关计算属性、模板结构和样式，避免留下空白占位或死代码。

## Goals / Non-Goals

**Goals:**

- 移除工作台首页和各业务页首屏的统计卡片模块，以及共享的工作区消息面板，让页面进入后优先呈现标题、操作和正文内容。
- 保持现有业务数据加载、按钮交互、状态标签和内容面板不受影响。
- 清理仅服务于头部辅助模块的前端计算属性、组件引用和样式定义，降低后续维护成本。
- 让移除后的页面在桌面端和移动端都保持自然间距，不出现突兀空白。

**Non-Goals:**

- 本次不重构页面路由、信息架构或导航方式。
- 本次不修改任何后端接口、工作区数据模型或推荐/监控算法。
- 本次不删除正文区域中承载业务明细的卡片、表格、时间线和预览内容。
- 本次不引入新的摘要替代组件或新的视觉体系。

## Decisions

### 1. 统一按“辅助模块容器”粒度移除，而不是逐个数字或消息替换文案

决策：直接删除首页 `metric-grid`、各业务页中的 `mini-stat-grid` 容器，以及壳层中的 `MessagePanel` 容器，而不是把数字卡片或消息条目逐个降级成纯文本提示。

原因：

- 用户诉求已经扩展为去掉这类页面进入后的辅助模块，按容器级移除更符合目标，也能避免留下半残的摘要区或空消息壳体。
- 这些卡片本质上是重复性汇总信息，不是页面主操作，删掉容器后页面结构更干净。
- 容器级删除实现更直接，测试边界也更清晰。

备选方案：

- 保留卡片或消息但降级为单行文本：仍然占据首屏注意力，没有真正解决信息噪音问题。
- 只隐藏数字或消息正文保留标题：会留下语义不完整的空壳模块。

### 2. 保留页头标题、操作按钮和轻量状态标签

决策：各页面继续保留 `panel-header`、按钮和已有的 `soft-tag` 等轻量状态信息，只删除数字摘要卡片。

原因：

- 标题和主操作是页面进入后的关键操作入口，删除它们会改变工作流而不只是精简展示。
- 一些状态标签直接服务当前操作，例如“处理中”“已确认条数”或最近更新时间，这类轻量信息不会形成统计模块负担。
- 这样可以把改动控制在用户明确要求的范围内，降低回归风险。

备选方案：

- 连同标签和说明一起大幅收缩页头：改动范围过大，容易引入新的可用性问题。

### 3. 同步删除仅为头部统计服务的前端派生状态和样式

决策：在模板删除统计模块后，继续清理不再被引用的 `computed` 汇总对象和 `main.css` 中仅对应统计卡片的样式规则。

原因：

- 如果只删模板，不删状态和样式，会留下未使用代码，后续维护时难以判断哪些字段仍有业务意义。
- 某些页面的 summary 计算只为卡片服务，保留它们没有价值，还会造成误导。
- 清理样式后可以避免响应式断点中仍然为已删除模块预留间距。

备选方案：

- 先只删模板、后续再清理：短期更快，但会制造技术债。

## Risks / Trade-offs

- [部分页面顶部信息被删后显得过空] → 通过保留标题、按钮、标签和原有正文面板，并在必要时微调间距避免塌陷。
- [删除汇总计算属性时误伤其他模板引用] → 先逐页核对绑定关系，再同步更新测试或构建校验。
- [样式或组件清理不彻底导致残留空白] → 同步检查 `metric-grid`、`mini-stat-grid`、`message-panel` 相关规则和 `MessagePanel` 引用。
- [用户实际期望包含更多首屏内容调整] → 先把本次变更收敛为“去掉统计模块”，其余视觉优化留到后续独立变更。

## Migration Plan

1. 先调整 `App.vue` 和各业务视图模板，删除统计卡片容器与共享消息面板并保留标题与主操作。
2. 清理对应的 `computed` 汇总状态、不再使用的组件引用和样式规则。
3. 运行前端测试或构建校验，确认页面仍可正常渲染，必要时补充/更新断言。
4. 若出现展示问题，可回滚本次前端模板与样式改动，后端和数据层无需迁移。

## Open Questions

- `App.vue` 侧边栏中的 `workspace-stats` 是否也应视为“头部统计模块”一并移除，还是保持现状，仅处理主内容区和各业务页首屏卡片。
- 轻量状态标签中如果继续保留数量信息，是否仍符合产品对“去掉统计模块”的预期，需要在实现阶段结合页面截图快速确认。
