## Context

当前仓库已经实现了项目经理 Agent 的核心离线能力，包括需求解析、模块知识库导入、分配推荐、确认、平台落地、Excel 同步、执行预警和团队洞察，但这些能力只通过 Python API 暴露。现有实现依赖调用方自行组织输入、保存中间结果并串联各阶段操作，不适合作为项目经理或组长的日常交互入口。

这次变更要解决的问题不是重新设计分配算法，而是在保留现有 `pm_agent` 领域逻辑的前提下，增加一个可直接访问的前后端分离 Web 工作台。由于当前仓库是纯 Python 项目，没有现成的前端工程、Node 工具链或 API 服务层，实现上需要兼顾两点：一是尽量复用已有工作流与状态模型；二是补齐前后端分离后的 API 边界、工作区状态和前端工程组织。

相关使用方包括项目经理、开发组长和测试负责人，他们需要在页面中完成四类高频动作：录入需求和成员信息、上传模块知识库与平台导出文件、查看并调整分配建议、查看同步结果与风险洞察。

## Goals / Non-Goals

**Goals:**

- 提供浏览器可访问的项目经理 Agent 工作台，覆盖需求录入、知识库导入、推荐生成、确认调整、平台落地预览、Excel 同步、执行预警和团队洞察。
- 让前后端分离架构下的 Vue 页面复用现有 `ProjectManagerAgent` 工作流与校验逻辑，避免复制业务规则。
- 引入工作区级状态管理，保存一次交互过程中的草稿输入、推荐结果和确认结果，支持页面刷新或分步操作后继续处理。
- 为页面动作定义清晰的 HTTP 接口和表单/文件上传协议，使 Vue 前端与后端 API 可以独立演进而不重写领域层。

**Non-Goals:**

- 本次不实现多用户权限、登录认证和组织级隔离。
- 本次不接入在线项目管理平台或 IM 系统，文件导入和手工输入仍是主要输入方式。
- 本次不改变现有需求分析、推荐、确认和监控的核心业务规则。
- 本次不追求通用设计系统或复杂可视化大屏，优先保证工作流完整可用。

## Decisions

### 1. 采用前后端分离架构，前端使用 Vue 单页应用

决策：新增后端 API 服务层和独立的 Vue 前端工程。前端负责页面路由、表单交互、列表编辑、文件上传和结果展示；后端负责工作区状态、文件接收、领域调用和结构化响应。

原因：

- 用户已经明确要求采用前后端分离并指定 Vue，设计必须以独立前端工程为前提。
- 需求录入、推荐确认、表格编辑和多步骤工作区天然适合用 SPA 处理，前端本地状态和交互反馈会明显优于服务端模板。
- 后端只保留 API 和领域适配职责，能继续稳定复用现有 Python 业务逻辑。

备选方案：

- 服务端模板渲染：基础设施更轻，但不符合用户指定架构，也不利于复杂工作区交互。
- 使用低代码或 Streamlit/Gradio：实现更快，但难以承载细粒度交互与长期演进。

### 2. 将前端流程组织为单工作区、多步骤的 Vue 工作台

决策：前端按一个工作区（workspace）组织，至少包含 `需求输入`、`成员与知识库`、`推荐结果`、`确认与落地`、`执行监控`、`团队洞察` 六个视图或页签，共享同一份工作上下文，并通过前端路由与状态管理统一承载。

原因：

- 当前领域流程是分步串行的，推荐、确认、同步和监控都依赖前一步的结果，工作区模型天然匹配现有 `ProjectManagerAgent` 的使用方式。
- 页面用户往往需要先录入需求，再补成员，再看推荐，再做确认；如果每步都变成独立页面且不共享上下文，体验会退化成“Web 版命令行”。
- 统一工作区后，页面可在任意步骤展示当前模块知识库、已确认需求数、最新导入批次和风险摘要。

备选方案：

- 每个动作独立成无状态页面：实现简单，但会导致大量重复输入和中间结果丢失。

### 3. 区分前端工作区状态与后端持久状态

决策：保留现有 `AgentState` 作为领域层持久状态，同时新增工作区状态对象。前端保存当前编辑态与界面筛选态，后端保存工作区草稿输入、最近一次标准化需求、推荐结果、确认动作和最近一次同步摘要，并通过 `workspace_id` 识别。

原因：

- 现有 `AgentState` 只保存模块知识库、确认结果、平台记录和导入批次，不包含页面草稿和推荐中间态，无法直接支撑工作台的多步骤交互。
- 前端工作区需要支持“生成推荐后再改派”“上传知识库后返回继续编辑需求”等场景，这些都要求保留中间状态。
- 文件级工作区持久化与当前 `LocalStateStore` 的实现方式一致，技术复杂度低，便于测试。

备选方案：

- 仅使用浏览器本地缓存保存草稿：实现快，但无法和后端状态联动，也不适合文件上传结果持久化。
- 直接把所有中间态塞进 `AgentState`：会污染现有领域模型，使 CLI/脚本调用也被迫理解页面概念。

### 4. 后端通过 API 应用服务适配现有 ProjectManagerAgent，而不是让控制器直接拼装领域调用

决策：新增 API 应用服务层，负责把 JSON 请求、表单字段和上传文件转换为 `ProjectManagerAgent` 可用的数据结构，并统一处理校验错误、工作区状态更新和接口响应模型构造。

原因：

- 当前 `ProjectManagerAgent` 方法已经具备清晰边界，适合被 API 层复用，但控制器直接调用会把请求映射、错误处理和状态保存逻辑散落在路由里。
- 应用服务层能把“录入需求并生成推荐”“提交确认并生成平台落地预览”“上传故事/任务导出并刷新监控摘要”封装成更高层动作，便于前后端联调和测试。

备选方案：

- 路由函数直接操作领域层：短期可行，但一旦接口动作增加，很快会失去可维护性。

### 5. 文件上传采用前端直传后端 API、服务端落盘 + 显式批次引用

决策：Vue 前端通过上传接口提交模块知识库、故事 Excel 和任务 Excel，后端将文件保存到工作目录下的上传区域，再由应用服务触发现有导入流程；前端只展示文件名、上传时间和导入结果摘要。

原因：

- 现有导入能力基于文件路径调用，服务端落盘可以最小化对领域层的修改。
- Excel 文件可能较大，前端不保留整份内容，只保留上传结果和批次摘要更稳妥。
- 保留文件路径和批次元数据有利于排查导入结果与复现实验。

备选方案：

- 将上传文件直接解析为内存对象传给领域层：减少磁盘 I/O，但需要大幅重写现有 Excel 导入入口。

### 6. 前端采用 Vue Router + 集中式状态管理，后端采用 REST 风格接口

决策：前端使用 Vue Router 组织工作区页面，并使用集中式状态管理保存当前工作区的草稿、推荐结果和查询状态；后端提供 REST 风格接口，区分工作区资源、推荐动作、确认动作和同步查询动作。

原因：

- 工作台天然是多视图、多步骤流程，路由和集中式状态管理可以避免组件间状态传递失控。
- 推荐生成、确认提交、文件上传和监控查询属于清晰的动作边界，REST 风格接口足以覆盖当前需求。
- 这套边界适合后续接入权限、审计日志或真正的多用户工作区能力。

备选方案：

- 前端全部依赖局部组件状态：初期可行，但在多页签和跨步骤回填场景下容易失控。
- 后端提供单一聚合接口：调用简单，但会让请求语义混乱，增加后续演进成本。

## Risks / Trade-offs

- [前后端分离增加工程和联调复杂度] → 通过明确 API 边界、工作区模型和分层测试控制复杂度，先覆盖主链路再扩展细节。
- [工作区状态与全局 AgentState 并存，可能出现同步理解偏差] → 明确区分“草稿/页面态”和“确认后持久态”，只在确认、知识库导入、平台同步等提交动作后回写领域状态。
- [文件上传会带来路径管理和清理问题] → 为每个工作区创建独立上传目录，并记录最近使用文件，后续通过定期清理策略处理历史文件。
- [单机文件存储不适合多用户并发] → 本次范围限定为本地或小规模内部使用，多用户与权限控制明确留到后续变更。
- [Vue 前端状态复杂度上升] → 只把跨页面共享的数据放入集中状态，局部表单态留在组件内部。
- [页面层可能掩盖底层校验细节] → 应用服务层统一透传字段级校验错误和业务阻塞信息，在前端显式展示，不把失败静默吞掉。

## Migration Plan

1. 新增后端 API 应用入口与基础资源路由，不影响现有 Python API 和测试。
2. 初始化 Vue 前端工程，搭建工作台骨架、路由和状态管理。
3. 引入工作区存储和应用服务层，先打通“需求录入 -> 推荐生成 -> 确认 -> 平台落地预览”主链路。
4. 增加模块知识库上传、故事/任务 Excel 同步、执行预警和团队洞察页面，并完成前后端联调。
5. 补充前端单元测试、后端接口测试和端到端流程测试后，再将 README 的典型用法扩展为“脚本调用 + 页面调用”双入口说明。

回滚策略：

- 如果前端或 API 层出现问题，可停止 Web 入口使用，现有 `pm_agent` Python API 与离线流程不受影响。
- 新增依赖和 Web 模块应保持与领域层松耦合，回滚时不需要迁移已有状态文件。

- 工作区是否只支持单用户“最近一次会话”，还是从第一版就允许创建多个并列工作区。
- 是否需要在页面中直接展示群回复文案预览，作为推荐结果的附加视图。
