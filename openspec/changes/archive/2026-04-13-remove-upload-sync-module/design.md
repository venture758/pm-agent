## Context

交付页当前包含三个 Tab（上传与同步、故事管理、任务管理）。其中“上传与同步”承担双文件同步入口，但该入口与故事管理页内的独立故事导入能力已产生功能重叠，且用户在日常操作中更常驻留于故事/任务管理视图。

本次设计目标是从 UI 层移除“上传与同步”模块，使交付页信息架构更聚焦，同时避免引入后端兼容性风险。

## Goals / Non-Goals

**Goals:**
- 在交付页移除“上传与同步”Tab及其内容区块。
- 将默认激活 Tab 调整为“故事管理”。
- 清理 upload-sync 相关的前端状态与模板代码，保证页面逻辑简化且可维护。
- 补充测试，确保两 Tab 导航和核心视图可用。

**Non-Goals:**
- 不在本次下线后端双文件同步 API。
- 不改动数据库模型。
- 不重构故事/任务列表数据结构。

## Decisions

### 1) 仅移除前端模块，不删除后端同步接口
- 决策：本次变更范围限制在前端页面结构和状态管理，后端 `/uploads/sync` 相关接口保持可用。
- 理由：该接口可能被脚本或历史流程使用，先做 UI 收敛可降低发布风险。
- 备选：同步删除后端接口与服务逻辑。
  - 未选原因：需要额外影响评估与迁移通知，不符合本次“删除页面模块”的最小变更目标。

### 2) 默认 Tab 改为 stories，保留 stories/tasks 两标签
- 决策：`activeTab` 默认值由 `upload-sync` 改为 `stories`，并移除 `upload-sync` 配置项。
- 理由：保证用户进入页面即看到有效管理数据，不再落在已删除模块。
- 备选：默认进入 tasks。
  - 未选原因：故事管理是主要入口，且已包含独立上传链路。

### 3) 删除 upload-sync 专属状态与方法
- 决策：删除仅服务于 upload-sync 区块的文件选择状态、同步按钮状态、同步完成提示与文案渲染。
- 理由：避免死代码和维护负担，防止后续逻辑误引用。
- 备选：保留状态但隐藏 UI。
  - 未选原因：会残留不可见行为，增加排查成本。

## Risks / Trade-offs

- [风险] 用户找不到历史“上传与同步”入口
  → Mitigation: 在故事管理中保留可用上传能力，必要时增加引导文案。
- [风险] 前端删除状态后引发回归（例如依赖变量未清理干净）
  → Mitigation: 补充 DeliveryView 的 Tab 渲染与交互测试，并执行前端构建验证。
- [权衡] 后端同步接口继续保留会产生短期“前后端能力不完全对称”
  → Mitigation: 后续可单独发起 API 下线提案，分阶段收敛。

## Migration Plan

1. 在 `DeliveryView.vue` 删除 upload-sync tab 定义与对应模板内容。
2. 调整默认 `activeTab` 为 `stories`，并清理 upload-sync 相关状态/方法。
3. 更新前端测试快照与行为断言（Tab 数量、默认选中、页面核心文案）。
4. 运行前端构建和相关测试后发布。

回滚策略：
- 若上线后用户反馈缺失关键入口，可回滚到上一版本恢复 upload-sync 模块；后端无需回滚。
