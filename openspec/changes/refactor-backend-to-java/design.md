## Context

当前系统后端以 Python 实现，已包含较完整的领域能力与 MySQL 存储结构，并由前端工作台直接调用。该变更目标是一次性替换为 Java 后端，同时允许 API 重设计并前后端同步改造。项目已明确技术与交付约束：
- 架构：模块化单体
- 技术栈：Java 21、Spring Boot、MyBatis、Maven
- 范围：全量能力（包含 chat 与 pipeline）
- 数据：历史数据全量迁移
- 上线：维护窗直切

## Goals / Non-Goals

**Goals:**
- 在 Java 中完整承接现有业务能力并统一领域边界与工程结构。
- 发布 `/api/v2` 版本化接口与统一响应模型，支撑前端同批次迁移。
- 通过版本化 SQL 脚本与迁移任务完成历史数据全量迁移，并形成可演练的切换流程。
- 保留 LLM 相关能力，包括 chat 需求解析、会话管理和 pipeline 分步推进。

**Non-Goals:**
- 不保留旧 Python API 的长期兼容层。
- 不在本次拆分微服务。
- 不在本次引入在线双写或长期并行运行。

## Decisions

### 1. 采用模块化单体而非微服务
- 选择：单服务部署，按业务域拆分模块（workspace、member、module、recommendation、confirmation、delivery、chat、pipeline、insight）。
- 原因：一次性替换窗口有限，优先保证重构可控、链路一致、发布简单。
- 备选方案：
  - 微服务拆分：边界清晰但迁移复杂度、跨服务事务和联调成本过高。
  - 简化平移单体：短期快，但无法收敛领域结构与技术债。

### 2. 采用 Spring Boot + MyBatis + 版本化 SQL 迁移脚本
- 选择：Spring Boot 提供成熟工程框架，MyBatis 保持 SQL 可控，版本化 SQL 脚本管理 schema 变更与迁移执行。
- 原因：现有 MySQL 表结构与查询较多，MyBatis 更适配复杂 SQL 与迁移过程中的显式控制。
- 备选方案：
  - JPA：开发效率高，但复杂查询与历史表兼容可控性较弱。
  - Liquibase：能力更强但配置成本更高，本项目优先执行效率。

### 3. API 使用版本化重设计并取消兼容层
- 选择：统一使用 `/api/v2/workspaces/{workspaceId}/...`，响应 envelope 固定为 `code/message/data`，分页字段统一。
- 原因：用户已接受前后端同步改造，可借此消除历史 API 负担并统一契约。
- 备选方案：增加 v1 适配层；会增加复杂度且违背一次性替换目标。

### 4. 全量迁移 + 维护窗直切
- 选择：切换前冻结写入，执行 schema 迁移 SQL 与迁移任务，校验通过后切流到 Java。
- 原因：用户明确选择一次性替换和维护窗直切，流程最直接。
- 备选方案：
  - 灰度切流：更稳妥，但需要双栈并行运维与更长周期。
  - 双写对账：一致性更强，但工程改造成本过高。

### 5. LLM 能力在 V1 保留并做分层降级
- 选择：抽象 LLM provider 层，支持 primary/fallback 重试和超时控制。
- 原因：chat 和 pipeline 是核心工作流，不能在 V1 缺失。
- 备选方案：V1 去除 LLM 或仅 mock，无法满足本次范围。

## Risks / Trade-offs

- [维护窗内迁移超时] -> 提前在预发做全量演练，固化耗时基线并提供分阶段校验点。
- [API 重设计导致前端联调阻塞] -> 先冻结并发布 OpenAPI 契约与 mock，前后端并行开发并按页面分批验收。
- [LLM 依赖不稳定] -> 统一超时、重试、降级响应与错误码，保障非 200 场景可观测。
- [数据迁移字段语义偏差] -> 建立表级行数校验 + 关键字段抽样校验 + 业务场景回放校验三层验证。
- [一次性替换回滚窗口小] -> 保留 Python 可回切版本、数据库快照和切换脚本，严格执行回滚门槛。

## Migration Plan

1. 完成 Java 工程骨架、schema 迁移 SQL 基线和 `/api/v2` 契约冻结。
2. 完成核心领域能力迁移与 LLM/pipeline 能力迁移，达到端到端可用。
3. 完成前端 API 客户端与页面适配，联调通过核心链路。
4. 在预发执行全量迁移演练（含时间统计、错误注入、回滚演练）。
5. 维护窗上线：冻结写入 -> 执行 schema 迁移 SQL + 数据迁移 -> 运行校验脚本 -> 切流 Java。
6. 上线后观察关键指标与错误告警，确认稳定后下线 Python 写流量。

回滚策略：
- 任一关键校验失败时停止切流并回滚到 Python 服务。
- 使用上线前数据库快照恢复到切换前状态，确保写入一致性。
- 回滚后保留迁移日志用于问题复盘和二次上线修正。

## Open Questions

- 当前阶段无阻塞性开放问题；如后续出现云资源或外部依赖限制，按“先满足维护窗切换稳定性”的原则调整非核心任务优先级。
